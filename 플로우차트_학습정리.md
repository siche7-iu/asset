# 플로우 차트(Flow Chart) 학습 정리

> 작성일: 2026-06-04
> 출처: Figma Resource Library — *Types of flow charts* (https://www.figma.com/ko-kr/resource-library/types-of-flow-charts/)
> 목적: 플로우차트 개념·유형·기호·작성법 학습 + **우리 고정자산관리 프로젝트에 어떤 흐름도를 그릴지** 준비.

---

## 1. 플로우차트란?

- **정의:** 프로세스·시스템·업무 흐름을 **표준 기호와 화살표**로 나타낸 시각 다이어그램.
- **목적:** 복잡한 프로세스를 지도처럼 그려, 순차적 정보를 팀·이해관계자에게 **명확하게 전달**. (특히 "복잡한 프로세스를 매핑하거나 시스템 상호작용을 시각화"할 때 유용)
- **핵심 가치:** 말·글로 설명하면 헷갈리는 절차를 "한눈에 보이게" 만든다.

---

## 2. 플로우차트 5대 구성 요소 (Figma 기준)

| 요소 | 설명 |
|------|------|
| **시작/종료점 (Start/End)** | 흐름의 명확한 시작과 끝 노드 |
| **하위 단계·처리 (Sub-steps/Processes)** | 흐름을 이끄는 순차적 작업들 |
| **방향 화살표 (Directional Arrows)** | 진행 순서와 관계 표시 |
| **분기점 (Decision Points)** | 여러 결과로 갈라지는 판단 지점 (예/아니오) |
| **표준화된 기호 (Standardized Symbols)** | 혼동 방지를 위해 일관된 기호 사용 |

---

## 3. 플로우차트 17가지 유형

> ✅ = 우리 자산관리 프로젝트에 바로 쓰기 좋음 / 🔶 = 상황에 따라 / ⬜ = 우리와 거리 있음

| 유형 | 용도 | 우리 적합도 |
|------|------|:---:|
| **Process Flow (프로세스 흐름도)** | 작업의 단계별 순서 + 분기점 | ✅ 자산 취득→감가상각→처분, 신청→승인 |
| **Workflow Diagram (워크플로우)** | 팀 업무 절차; 온보딩·프로세스 개선 | ✅ 부서별 자산 업무 흐름 |
| **Swimlane Diagram (스윔레인)** | 여러 부서/역할의 책임 분담 표시 | ✅ 신청자·검증자·결재자 역할 구분 |
| **Decision Tree (의사결정 트리)** | 선택에 따른 결과 분석 | ✅ 노후/불용 판정, 예외 승인 분기 |
| **Data Flow Diagram (데이터 흐름도)** | 데이터가 시스템을 따라 이동·변환되는 경로 | 🔶 외부 API→DB→화면 데이터 흐름 |
| **System Flowchart (시스템 흐름도)** | 외부 입력이 시스템 처리를 유발하는 구조 | 🔶 외부 연계(GSE/MCA/API) 구조 |
| **Document Flow (문서 흐름도)** | 결재 단계와 필요 서명자 추적 | ✅ 결재선·감사 대응 문서 흐름 |
| **Use Case Diagram (유스케이스)** | 사용자가 시스템과 상호작용하는 방식 | 🔶 AI Agent 사용 시나리오 |
| **Customer Journey (고객 여정)** | 발견→구매까지 사용자 진행 | 🔶 사용자(직원) 업무 여정 |
| **Logic Model (로직 모델)** | 입력→처리→장기 성과 문서화 | ⬜ |
| **Event Storming** | 프로그램 실행에 필요한 이벤트 매핑 | ⬜ |
| **PERT Chart** | 프로젝트 일정·의존관계 관리 | ⬜ (사업 일정엔 가능) |
| **Code Flowchart (코드 흐름도)** | 프로그램 로직·조건 분기 | ⬜ |
| **Product Flow / Production Flow** | 제조·공급망 / 주문 이행 단계 | ⬜ |
| **E-commerce Flow** | 온라인 판매 파이프라인 | ⬜ |
| **Website Flowchart** | 사이트 구조·페이지 연결 | 🔶 화면 전환(SPA 라우팅) 맵 |

## 4. 표준 기호(도형) — 실제로 그릴 때 쓰는 모양

> Figma 자료는 "표준 기호를 일관되게 쓰라"고 강조. 아래는 통용되는 표준 플로우차트 기호.

| 도형 | 이름 | 의미 |
|------|------|------|
| 타원/둥근 사각 ⬭ | **터미네이터(Terminator)** | 시작 / 종료 |
| 직사각형 ▭ | **프로세스(Process)** | 처리·작업 단계 |
| 마름모 ◇ | **판단(Decision)** | 조건 분기 (예/아니오) |
| 평행사변형 ▱ | **입출력(Input/Output)** | 데이터 입력·출력 |
| 양옆 줄 직사각형 | **사전정의 프로세스** | 미리 정의된 하위 절차(서브루틴) |
| 문서 모양 | **문서(Document)** | 보고서·서류·출력물 |
| 원통 | **데이터베이스(Database)** | 데이터 저장소 |
| 작은 원 ◯ | **연결자(Connector)** | 흐름 연결(페이지 내/간) |
| 화살표 → | **흐름선(Flow line)** | 진행 방향 |

## 5. 작성 Best Practice (Figma 권장)

1. 명확한 **시작점**에서 출발한다.
2. 시스템 결과를 보여주는 **명시적 종료점**을 넣는다.
3. 기호는 **화살표로 연결**해 방향을 분명히 한다.
4. 텍스트는 **짧고 핵심만**.
5. 단계는 **논리적 순서**로 진행한다.
6. **일관된 표준 기호**를 끝까지 동일하게 적용한다.
7. 관찰/프로토타입 모드로 **공유해 피드백** 받는다.
8. **실시간 협업**으로 수정한다.

---

## 6. 작성 단계 (Figma 권장 프로세스)

1. **범위 정의** — 시각화할 프로세스/시스템 결정
2. **단계 매핑** — 복잡한 흐름을 구성 단계로 분해
3. **분기점 식별** — 분기 로직이 생기는 지점 표시
4. **기호 적용** — 요소별 표준 도형 배정
5. **방향 흐름 추가** — 화살표로 순서 연결
6. **검토** — 일관된 구조인지 확인
7. **협업** — FigJam 등으로 공유·피드백
8. **개선** — 피드백 반영해 다듬기
> ※ FigJam 템플릿은 "표준 기호가 미리 만들어져 있어 시간 절약" 가능.

## 7. 우리 프로젝트 적용 메모 (다음에 그릴 후보)

> 실제 작도는 별도 요청 시 진행. RFP·제안서(`제안서_Asis_정리.md`)에서 "단계별 현황판·프로세스 재설계"가 핵심 키워드였으므로, 아래 흐름도들이 시연·설계에 유용.

| 그릴 후보 | 추천 유형 | 근거(RFP/제안서) |
|----------|----------|------------------|
| **자산 생애주기 흐름** (취득→운영→감가상각→처분/폐기) | Process Flow | 제안서 1.1.2·1.5.2 |
| **자산 신청→검증→결재→승인→사후관리** | Swimlane(역할별) + Process | 제안서 1.4.1, RFP 자산별 프로세스 |
| **외부 API 자동 검증 흐름** (입력→시세/지도/인사DB 조회→적정성 판정) | Decision Tree / Data Flow | 제안서 1.4.2 |
| **AI Agent 동작 흐름** (질의→데이터 조회→정책 검토→답변→보고서) | Process Flow / Use Case | 제안서 1.2, `AI_AGENT_제안분석.md` |
| **예산 흐름** (편성→배정→집행→환수/재배정) | Process Flow | 제안서 1.3.1 |
| **계약(사무) 프로세스** (수요발생→입찰공고→제안서접수→평가/낙찰→계약체결) | Process Flow + 체크리스트 | 제안서 1.6.2 |
| **화면 전환 맵** (인트로→대시보드↔목록↔상세↔AI Agent↔보고서) | Website Flowchart | 우리 프로토타입 SPA 라우팅 |

> 도구 옵션: ① Figma/FigJam에서 직접 작도, ② 코드로 그리려면 우리 프로젝트 무라이브러리 원칙에 맞춰 **SVG/CSS 또는 Mermaid 텍스트**로 표현 가능. (선택은 작도 요청 시 결정)
