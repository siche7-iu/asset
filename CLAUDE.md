## 프로젝트 개요

- **무엇**: 고정자산관리시스템(Fixed Asset Management System)의 **시연용 프로토타입**.
- **목적**: 실제 운영 시스템이 아니라, 시연(데모)으로 보여주기 위한 화면을 만드는 것이 목표.
- **실제 개발**: 운영 버전은 별도의 내부 보안 환경에서 진행됨. 따라서 이 프로토타입은 "운영급 구조"보다 "쉽게 실행하고 보여주기"를 우선한다.
- **사용자**: 개발에 익숙하지 않은 기획자. 설명은 한국어로, 전문 용어를 최소화한다. 큰 작업은 작은 단위로 쪼개서 진행한다.

## 기술 구성 (의도적으로 단순하게)

- 순수 **HTML + CSS + JavaScript**만 사용한다. 외부 라이브러리·빌드 도구·서버가 **전혀 필요 없다**.
- 차트는 **Chart.js**(일반 업무 차트)와 **Plotly.js**(AI 분석 차트)를 `js/lib/`에 로컬 저장해 사용한다(인터넷 없이도 동작). 기존 대시보드는 SVG 직접 그리기 방식 유지.
- 자산 데이터는 `js/data.js`에 코드로 직접 넣어 둔 샘플 데이터다(데이터베이스 없음). 매번 같은 데이터가 보이도록 무작위 없이 규칙으로 생성한다.

## 실행 방법

- 설치할 것이 없다. **`index.html` 파일을 더블클릭**하면 기본 웹 브라우저에서 바로 열린다.
- (선택) 명령으로 열기: PowerShell에서 `start index.html`
- 코드를 고친 뒤에는 브라우저에서 **새로고침(F5)** 하면 바뀐 내용이 반영된다.

### 동작 방식(큰 그림)

- 화면은 페이지 이동 없이 한 페이지 안에서 **세 개의 영역을 보였다 숨겼다** 하며 전환한다.
  - `#view-dashboard` (대시보드) · `#view-list` (자산 목록) · `#view-detail` (자산 상세)
  - 왼쪽 메뉴 버튼(`.nav-item`)이 `showView()`를 호출해 전환한다.
- `js/app.js`는 시작할 때 `window.APP_DATA.assets`(자산 목록)를 읽어:
  - **대시보드**: 합계/개수를 계산하고, 분류별·부서별은 막대그래프, 상태별은 도넛 그래프로 그린다.
  - **목록**: 표로 그리고, 검색어와 분류 선택으로 걸러서 다시 그린다.
  - **상세**: 목록의 행을 클릭하면 해당 자산의 정보와 변경 이력(타임라인)을 보여준다.
- 자산 한 건의 데이터 형태(`js/data.js`):
  `id`(자산번호), `name`(자산명), `category`(분류), `model`(모델), `department`(사용부서),
  `owner`(사용자), `location`(위치), `status`(상태: 사용중/유휴/수리중/폐기예정),
  `acquireDate`(취득일), `price`(취득금액), `history`(변경 이력 배열: `date`, `type`, `detail`).

## 작업 규칙 (중요)

- 모든 대화·설명은 **한국어**로, 전문 용어는 최소화한다.
- **문장을 작성할 때는 사실과 논리에 맞게 작성한다.** 기술 용어·개념·동작 방식을 설명할 때 틀린 표현이나 논리적으로 어긋난 서술을 하지 않는다. 확신이 없으면 모호하게 넘기지 말고 정확한 표현으로 고쳐 쓴다.
- 큰 작업은 **작은 단위로 쪼개서** 진행한다.
- **작업을 할 때마다 이 파일(`CLAUDE.md`) 맨 아래 "작업 로그"에 한 줄로 간단히 기록**한다(날짜 + 무엇을 했는지, 쉬운 말로).
- **기본 모델은 Sonnet을 쓴다.** 빠르고 비용 효율적인 Sonnet을 기본으로 사용한다. 아주 복잡한 설계·깊은 분석·어려운 디버깅 등 **꼭 필요한 경우에만** 그 작업에 한해 Opus를 쓰고(주로 Opus 보조 에이전트에게 위임하는 방식), 끝나면 다시 Sonnet으로 돌아온다.
- **요구사항정의서 Excel 파일(`01.요구사항정의서_v2_작성중_260609.xls`) 관련 작업을 할 때는, 작업 시작 전에 반드시 `D:\업무\고정자산관리시스템_재구축\요구사항정의\요구사항정의_작업노트.md`를 먼저 읽는다.** 컬럼 매핑·RFP 분류 매핑·반영여부 기준·스크립트 위치 등 모든 작업 기준이 이 파일에 있다. 단, 이 Excel 작업은 프로토타입 사이트의 `js/app.js` `REQ_DATA`(50건)와 **완전히 별개**이므로 절대 혼용하지 않는다.
- **.xls 파일 작업 요청 시, 아래 조건 중 하나라도 해당하면 "`.xlsx`로 전환하면 Python으로 더 쉽게 처리할 수 있다"고 먼저 알리고 전환 여부를 물어본 뒤 진행한다.** 전환하지 않겠다고 하면 기존 PowerShell COM 방식으로 진행한다.
  - 데이터를 **읽어서 필터링·집계·분석**하는 작업
  - 여러 시트나 여러 파일을 **합치거나 비교**하는 작업
  - 특정 조건에 맞는 행만 **골라서 수정**하는 작업
  - 스크립트를 **반복 재사용**할 가능성이 높은 작업

- **자율모드는 기본값이다.** 대화 시작부터 자율모드로 동작한다.
- 파일 생성·수정·명령 실행 시 "진행할까요?" 같은 확인 질문 없이 바로 실행한다.
- 결과만 간결하게 보고한다. 방법론·계획·확인 질문은 생략한다.
- **반드시 멈추고 확인을 받는 경우** (아래 중 하나라도 해당하면 실행 전에 반드시 묻는다):
  - 파일 삭제 또는 덮어쓰기
  - `git push` · 배포(Vercel) 등 외부 영향 작업은 **기본적으로 알아서 실행**하되, 중요하거나 위험성이 있어 직접 판단이 필요하다고 판단될 때만 먼저 확인한다
  - 작업 범위가 3개 파일 이상 동시에 바뀌는 대규모 변경
  - 되돌리기 어렵다고 판단되는 모든 작업
  - 방향 자체가 불명확해 어떤 판단도 불가능할 때

### 팀장/팀원 역할 분리 (항상 준수)

> **핵심 원칙**: 사용자(나)에게 직접 터미널에서 지시를 받는 에이전트는 **무조건 처음부터 팀장으로 기동**한다. 팀장은 직접 작업을 수행하지 않고, **반드시 하위 팀원(Agent)에게 작업을 오케스트레이션하여 진행**한다. 팀장이 직접 작업에 몰두하면 사용자가 중간에 지시를 내릴 수 없으므로, 팀장은 항상 대화 준비 상태를 유지해야 한다.

**팀장(Claude 메인)의 역할**
- 사용자 지시 수신 → 작업 분석 → 팀원에게 배분
- 진행 상황 모니터링 및 중간 보고
- 팀원 결과물 검증 (코드 리뷰·동작 확인)
- CLAUDE.md 작업 로그·TODO·HANDOFF 문서 직접 업데이트
- git commit·push (문서·메타 파일에 한해서만 직접 수행)
- 사용자 질문에 즉시 응답 (작업 진행 중에도 항상 대화 가능)

**팀원(Agent)의 역할**
- `executor` — HTML/CSS/JS 코드 작성·파일 수정 (대부분의 구현 작업)
- `Explore` — 코드·파일 탐색 및 구조 파악
- `code-reviewer` — 완성된 코드 검토·품질 검증
- `architect` — 복잡한 설계·구조 분석 (Opus 사용)
- `debugger` — 버그 원인 추적·수정

**위임 방법**
- 팀원에게 지시할 때는 **무엇을·어느 파일에·어떻게** 를 명확히 전달한다.
- 팀원은 `run_in_background: false` (결과를 받아야 할 때)로 기본 실행한다.
- 독립적인 여러 작업은 **동시에 여러 팀원에게** 병렬 위임한다.
- 팀원 작업이 끝나면 팀장이 결과를 검토한 뒤 사용자에게 보고한다.

**팀장이 직접 해도 되는 것** (가볍고 즉각적인 메타 작업)
- CLAUDE.md·HANDOFF.md·TODO.md 문서 수정
- git status·git log 확인
- 파일 삭제 등 단순 명령 1줄짜리 작업
- 사용자에게 현황 브리핑·질문 답변

### 플러그인 자동 실행 규칙

#### `/feature-dev` — 신규 기능 개발 워크플로우

> 7단계(탐색→질문→설계→구현→품질 리뷰→요약) 구조화 워크플로우. **명령어를 직접 실행**한다.

#### `/ast-grep` — AST 패턴 코드 검색 (커스텀 커맨드)

> `sg` CLI를 사용한 구조적 코드 탐색. 텍스트 grep이 아닌 **코드 문법을 이해하는 패턴 검색**. **별도 명령 없이 자동 활성화**된다.

**자동 활성화 조건:**
- **함수·이벤트 핸들러·DOM 조작 패턴** 등을 코드 구조 기준으로 찾을 때
- 리팩터링 전 특정 패턴의 **전체 사용처 파악**이 필요할 때
- 단순 문자열 grep으로는 **정확도가 부족한** 코드 탐색 작업

**사용법:** `sg --pattern '<패턴>' --lang js js/` (이 프로젝트 기본)
**전제 조건:** `npm install -g @ast-grep/cli` 완료 (`sg` 0.43.0 설치됨)
**사용하지 않는 경우:** 단순 문자열 검색(Grep 도구로 충분)

### 세션 시작 브리핑 (그날 첫 대화에서만)

- 대화가 새로 시작될 때, 첫 응답에서 TODO.md·HANDOFF.md·작업 로그를 읽고 **"오늘 뭐부터 하면 좋을지" 3줄 이내로 간단히 브리핑**한다.
- 브리핑 후 바로 지시를 기다린다. 허락 없이 먼저 실행하지 않는다.
- 같은 세션 안에서 두 번째 대화부터는 브리핑을 반복하지 않는다.

### 개발 원칙 (코드를 작성·수정할 때)

1. **먼저 주변을 본다.** 코드를 작성하거나 고치기 전에, 반드시 **고칠 부분의 주변 코드 구조를 먼저 분석**한다. (어떤 함수·변수·규칙을 따라야 하는지 파악한 뒤에 손댄다.)
2. **예외 상황부터 적는다.** 해결책을 바로 내놓지 말고, 발생할 수 있는 **예외 상황(Edge Case)을 최소 3가지 이상 먼저 목록으로** 정리한다. (예: 데이터가 비었을 때, 값이 너무 클 때, 예상과 다른 형식이 들어올 때 등.)
3. **끝나면 스스로 검토하고 보고한다.** 코드가 완성되면 **스스로 코드 리뷰**를 하고, **보안·성능상 문제가 없는지 검증한 결과를 쉬운 말로 요약해서 보고**한다.
4. **요구사항 데이터는 두 곳을 항상 동시에 수정한다.** `js/app.js`의 `REQ_DATA`(그리드·CSV의 원본)와 `요구사항_상세정의.md`(전체 요구사항 보기 문서)는 1대1로 대응한다. 한쪽을 수정하면 반드시 다른 쪽도 같은 내용으로 맞춰야 하며, 둘이 달라지면 안 된다.

## Tooling for Shell Interactions

- 예시: `ast-grep --lang typescript -p '$A && $A()'`
