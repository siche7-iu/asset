# 비밀 프로젝트 화면 — 비밀번호 관리 안내 (HASHGEN)

> **대상**: 팀장 · 이 프로젝트를 이어받는 에이전트/개발자
> **요약**: 사이드바 `v0.9.3 · NH Demo Build` 클릭 → 비밀번호 입력 → 프로젝트 관리 화면(`view-project`) 진입.
> 비밀번호 **평문을 소스에 두지 않고 SHA-256 해시값만** 저장한다.

---

## 1. 무엇을 바꿨나 (2026-06-04)

- **이전**: `js/app.js`에 `var PW_CODE = "78963";` — 비밀번호가 소스/개발자도구/소스보기에 **그대로 노출**.
- **이후**: 비밀번호 평문 제거. **SHA-256 해시값(`PW_HASH`)만 저장**하고, 입력값을 `SALT+입력`으로 해시해 비교.
  - 소스에는 의미 없는 64자리 16진수 해시만 보임 → **원문(78963) 복구 불가(단방향)**.
  - 외부 라이브러리·`crypto.subtle` 없이 **순수 JS SHA-256** 사용 → 인터넷·서버 없이 `index.html` 더블클릭(`file://`)에서도 동작.
- **현재 비밀번호**: 기존과 동일 (`78963`). 기능·동작 100% 동일, Playwright 테스트 5건 통과(`tests/project-secret.spec.js`).

관련 코드: `js/app.js`의 `// ===== 비밀 버튼 → 프로젝트 관리 모드 =====` 블록
- `PW_SALT` : 해시에 섞는 고정 문자열 (비밀 아님, 레인보우테이블 방지용)
- `PW_HASH` : `sha256(PW_SALT + 비밀번호)` 결과. **이 값만 바꾸면 비밀번호가 바뀐다.**
- `sha256()` : 표준과 동일한 결과를 내는 순수 JS 구현
- `submitPw()` : `sha256(PW_SALT + 입력) === PW_HASH` 로 검증

## 2. 비밀번호 바꾸는 법 (해시 재생성)

`PW_SALT`는 그대로 두고, 새 비밀번호로 **새 해시값**을 만들어 `js/app.js`의 `PW_HASH`만 교체하면 된다.

### 방법 A — PowerShell (설치 불필요)
새 비밀번호가 예를 들어 `mySecret!`라면:
```powershell
$salt = "nh-fams-proj::"
$pw   = "mySecret!"          # ← 새 비밀번호로 변경
$bytes = [Text.Encoding]::UTF8.GetBytes($salt + $pw)
$sha = [Security.Cryptography.SHA256]::Create()
-join ($sha.ComputeHash($bytes) | ForEach-Object { $_.ToString("x2") })
```
출력된 64자리 문자열을 `js/app.js`의 `PW_HASH = "...";` 에 붙여넣는다.

### 방법 B — Node.js
```bash
node -e "console.log(require('crypto').createHash('sha256').update('nh-fams-proj::'+'mySecret!').digest('hex'))"
```
(`'mySecret!'` 부분만 새 비밀번호로 교체)

> ⚠️ `PW_SALT`를 바꾸면 해시도 그 salt로 다시 만들어야 한다. salt와 해시는 항상 짝.

## 3. 보안 수준 — 솔직한 한계 (꼭 읽을 것)

- ✅ **달성**: 비밀번호 글자 자체는 소스/개발자도구/소스보기 어디에도 안 보인다. 해시는 단방향이라 역산 불가.
- ⚠️ **한계 1 (원리상)**: 순수 프론트엔드(서버 없음)에서는 검증 로직이 모두 브라우저에 있으므로, 작정하면 누구나 분석할 수 있다. **"완벽한 비밀"은 서버 인증 없이는 불가능**하다.
- ⚠️ **한계 2 (무차별 대입)**: 현재 비밀번호 `78963`은 **5자리 숫자(10만 가지)**라, 해시를 가진 사람이 전수 대입하면 금방 뚫린다.
  - 진짜로 보호가 필요하면 **길고 영문·숫자·기호가 섞인 비밀번호**로 바꿀 것(방법 2 참고). 그래야 무차별 대입이 사실상 불가능해진다.
- 📌 **결론**: 이 잠금은 *"시연 중 우연히/소스 슬쩍 봐서 노출되는 것"을 막는 용도*로는 충분. *"민감정보를 외부 공격자로부터 지키는 용도"*라면 별도 서버 인증이 필요하다.
