## Q1. 시스템 모니터링 도구 보유 여부

**영역**: 시스템 현황  
**질의**: 시스템 모니터링 도구 보유 여부 확인 요청 (제니퍼, 스카우트 등).  
운영 시스템의 CPU 사용율·메모리 사용율·TPS 등 모니터링 여부 — 안정화 지원 목적.

**NH 답변**: 사내 통합관제시스템을 통해 모니터링 가능합니다.

**PM 해석**: 운영 시스템 반영 후 안정성 모니터링을 위한 질문이었고, 모니터링 툴 기능에 따라 효과가 달라질 수 있어서 질문함. **특이사항 없음.**

## Q2. 모바일 App(NH재물조사) 개발 범위 포함 여부

**영역**: 개발 범위  
**질의**: 프로그램 수행 범위에 NH재물조사 모바일 App 영역의 대응 개발 포함 여부 확인.

**NH 답변**:
- 현 고정자산시스템 내 인터페이스 연결 등의 기능 전환 개발 포함.
- 단, 별도의 업무 프로세스 제안이 필요한 경우 해당 기능은 제안서에 포함 **필수**.

**PM 해석**:
- 자산관리시스템 기능 변경 시 모바일 앱 화면/소스 수정이 사업 범위에 포함되는지 확인 목적.
1. 모바일 App 기능 개선 시 자산관리시스템 적용은 수행 범위 (SR 대응 형식). 모바일 APP이 먼저 바뀌는 케이스가 있는지 추가 확인 필요.
2. 개선 요건 정의 시 모바일 앱 영향도를 항시 점검 필요.
3. 모바일 APP 영역 일부 수정이 필요한 경우 대응 개발자 투입 여부 확인 필요. (앱 수정이 아닌 모바일 Web 개발 대응 범위로 처리될 가능성.)

## Q3. 형상관리 도구 현황

**영역**: 아키텍처  
**질의**: 현재 운영 중인 형상관리 툴 (SVN·GitLab 등), 화면/AP 동일 도구 적용 여부 확인.

**NH 답변**:
- 화면(UI): SVN 사용하여 업무 담당자 간 소스 공유 중.
- AP 소스: 사내 개발서버망을 통해 화면 소스 및 AP 소스 최종 관리 중. **(별도 툴 없음)**

**PM 해석**: 고객 형상관리는 **SVN 사용 중**. 특이사항 없음.

## Q4. 보고서/리포트 도구 및 보고서 수량

**영역**: 보고서/솔루션  
**질의**:
1. 마이빌더 → 웹스퀘어 전환 시 사용 가능한 Report Tool (ReportDesigner 등) 유무 확인.
2. 현재 사용 중인 보고서 파일의 종류 및 수량.

**NH 답변**:
1. 현재 마이빌더의 리포트 기능 사용 중, **별도 툴 없음**. 이전 미팅에서 PDF 변환 방향으로 개발되어야 한다고 언급됨.
2. 현재 **보고서 파일 180개** (마이빌더 mrf 양식).

**PM 해석** ⚠️ **리스크 주의**:
- 웹스퀘어에는 자체 보고서 리포트 기능이 없음.
- 보고서를 화면(퍼블리싱)으로 개발하고 오픈소스 lib 연계하여 PDF 변환하는 방식으로 구현한 사례 있으나, **보고서가 180개로 많아** 연계 가능한 별도 Report Tool 확인 요청 필요.
- 보고서가 수십~수백 장 출력되는 경우 구현 난이도 상승 + 유지보수 어려움 예상.
- **180개 보고서 신규로 모두 개발해야 하는 문제 존재.**

## Q5. AsIs 시스템 통신 방식 및 AP 서버 구성

**영역**: 시스템 현황  
**질의**:
1. AsIs 시스템에서 마이빌더 ~ AP 간 통신 시 메시지 구조 (XML, JSON 등).
2. AP서버가 ProFrame C로 구성되어 있는 것으로 알고 있음. ProFrame에서 JSON 방식 거래 가능 여부 확인 (WebSquare 표준이 JSON 방식).

**NH 답변**:
1. WAS(JEUS)에서 **TMAX 서비스 전문 호출** 방식으로 통신.
2. **pro\*C**를 사용하여 개발되어 있으며, 프로프레임(프레임워크)은 사용하지 않음. 해당 부분은 미팅 후 추가 확인하여 연락 예정.

**PM 해석** ⚠️ **공통 개발자 투입 필요**:
- 별도 시스템 공통 개발자를 투입할 필요는 없겠지만, **투입된 공통 개발자가 웹스퀘어 F/W ~ WAS(JEUS) 간 통신 가능한 모듈/서비스를 개발**해야 함.
- 공통 개발자 역할 및 책임 범위 사전 정의 필요.
