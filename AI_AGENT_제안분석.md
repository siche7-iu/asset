## 0. 큰 그림 — 이 3장은 한 세트다 (3계층 구조)

이 세 장은 따로 노는 화면이 아니라, **하나의 기업용 생성형 AI 플랫폼을 위→아래 3개 층으로 쪼갠 것**입니다.

```
┌─────────────────────────────────────────────┐
│  ① 2.3 AI 서비스 플랫폼   (사용자/현업이 쓰는 층)   │  ← 사람이 만지는 화면
│     "AI를 만들고 쓰는 곳"                          │
├─────────────────────────────────────────────┤
│  ② 2.3.3 Multi Agent FW  (여러 AI가 협업하는 층)  │  ← AI들의 역할/권한/연동
│     "여러 전문 AI가 손발 맞추는 곳"                  │
├─────────────────────────────────────────────┤
│  ③ 2.4 AI 모델 영역(LLM Ops) (엔진/인프라 층)      │  ← GPU·모델 운영 (백엔드)
│     "AI의 엔진(모델)을 관리·배포·검증하는 곳"          │
└─────────────────────────────────────────────┘
```

**한 줄 요약(발표용):**
"위층은 직원이 AI를 직접 만들고 쓰는 화면, 가운데층은 여러 전문 AI가 협업하도록 묶는 틀, 아래층은 그 AI들의 엔진(모델·GPU)을 운영·검증하는 관제실입니다."

## 1. 〔이미지 ①〕 2.3 AI 서비스 플랫폼 — "AI를 만들고 쓰는 곳"

현업 직원·개발자가 **코딩 없이(No-Code/Low-Code) AI 에이전트를 만들고, 쓰고, 점검**하는 화면 모음. 12개 영역으로 구성.

| # | 영역(화면) | 무엇을 보여주나 | 발표 포인트 |
|---|-----------|----------------|------------|
| 1 | **MCP 추가/삭제·승인/이력 관리** | 외부 도구·데이터를 AI에 연결하는 "MCP"를 No-Code로 등록·삭제하고, 변경 이력을 남김 | "AI에게 새 도구(손)를 끼워주는 곳. 누가 언제 바꿨는지 기록 남김" |
| 2 | **서버 목록 관리 & 앱스토어** | 등록된 MCP/Agent를 목록·앱스토어 형태로 공유 | "잘 만든 AI 기능을 사내 앱스토어처럼 나눠 씀" |
| 3 | **단일 Agent 개발** | 플랫폼 기본 기능만으로 AI 하나를 제작 | "코딩 없이 AI 하나 뚝딱 만들기" |
| 4 | **자연어 처리용 Chat UI** | 일반 채팅창처럼 말로 시키는 화면 | "직원은 그냥 말로 시키면 됨 (우리 AI Agent 화면과 동일 컨셉)" |
| 5 | **개별 에이전트 활용 + FLOW UI 구성** | 사용자가 자기 AI를 FLOW(순서도) 화면으로 구성 | "AI가 일하는 순서를 그림(블록)으로 짠다" |
| 6 | **FLOW UI 액션 블록 (Drag & Drop)** | 기능 블록을 끌어다 놓아 AI 업무 흐름 제작 | "레고처럼 블록을 드래그&드롭해 자동화 흐름 완성" |
| 7 | **가상환경 내 SDK 활용** | 개발자는 SDK(코드)로 더 깊게 커스터마이즈 | "고급 사용자는 코드로도 가능" |
| 8 | **사용자/프로젝트별 권한 관리** | 누가 어떤 AI·문서에 접근 가능한지 권한 부여 | "보안: 부서·사람별로 접근 제어" |
| 9 | **Datumo Eval — 평가 Dashboard** | AI 답변 신뢰성을 BAD/MODERATE/GOOD/EXCELLENT로 점수화 | "AI 답이 믿을 만한지 자동 채점 (국내 최초 All-in-one 검증)" |
| 10 | **자동 평가로 모델 검증/비교** | 여러 모델을 같은 기준으로 비교 (지표: Content Relevance, Response Relevancy, Faithfulness/할루시네이션) | "어느 AI가 더 정확한지 객관 비교" |
| 11 | **Agent 컴포넌트 모니터링** | Agent 사용량·LLM 호출·상태를 대시보드로 감시 | "어떤 AI가 얼마나 쓰이는지 관제" |
| 12 | **외부 모델 반입 프로세스** | 외부 AI 모델을 들여올 때 책임자 승인 단계(요청→사용→선택→확인→최종승인) | "보안망 안으로 외부 AI 들이는 결재 절차" |

**핵심 메시지:** "현업이 코딩 없이 AI를 만들고(빌더·FLOW), 말로 쓰고(Chat), 믿을 만한지 채점하고(Datumo Eval), 안전하게 관리(권한·승인)하는 통합 작업장."

## 2. 〔이미지 ②〕 2.3.3 Multi Agent FW — "여러 AI가 협업하는 틀"

AI 하나가 아니라 **여러 전문 AI(에이전트)가 역할을 나눠 손발을 맞추는** 구조. 7개 영역.

| # | 영역(화면) | 무엇을 보여주나 | 발표 포인트 |
|---|-----------|----------------|------------|
| 1 | **자원 권한 부여·관리 (1/3)** | Agent·문서 등을 사용자/프로젝트/그룹별로 접근 권한 설정 | "각 AI가 건드릴 수 있는 자료를 부서별로 제한" |
| 2 | **Agent 역할 정의 (2/3)** | AI 하나의 정체성 설정: 설명·LLM 모델·지침(프롬프트)·연결 도구(Tool) | "이 AI는 누구이고, 어떤 모델로, 무슨 지침으로 일하는지 정의" |
| 3 | **Agent 상호 연동·노드 전달 (3/3)** | 한 AI의 결과를 다음 AI의 입력으로 넘기는 연결(노드) | "AI A의 결과 → AI B의 입력으로 바통 터치" |
| 4 | **MCP 유기적 연계 (리소스 관리 구성도)** | 기본 MCP + Datumo Eval + 외부 API(예: 금융감독원 DART)를 묶어 도구화 | "여러 도구를 한 다발로 묶어 AI가 골라 쓰게" |
| 5 | **그룹/멤버/권한별 Agent 추가·삭제** | Agent 생성·수정·공개 권한 및 정책 재설정 | "AI를 늘리고 줄일 때 권한·정책 자동 정리" |
| 6 | **배포 전 시뮬레이션·이행 체계** | Low-Code로 만든 흐름을 배포 전에 즉시 테스트 | "실제 투입 전에 리허설" |
| 7 | **No-Code Agent 테스트** | 배포 후 실제 환경 응답 확인·재검증·실행 이력 | "투입 후에도 잘 도는지 계속 점검" |

**핵심 메시지:** "전문 분야가 다른 AI들을 ①역할 정의 → ②권한 부여 → ③서로 연결(노드 전달) → ④배포 전후 테스트하는, '여러 AI를 한 팀처럼 운영'하는 틀."

## 3. 〔이미지 ③〕 2.4 AI 모델 영역(LLM Ops) — "엔진·인프라 관제실"

AI의 엔진인 **모델과 GPU 자원을 운영하는 백엔드 층**. 우리 같은 시연 프로토타입과는 **가장 거리가 먼(=참고만)** 영역. 7개 영역.

| # | 영역(화면) | 무엇을 보여주나 | 발표 포인트 |
|---|-----------|----------------|------------|
| 1 | **통합 모델 카탈로그** | 내/외부 모델을 등록·검색·버전관리 | "쓸 수 있는 AI 모델들의 도서관" |
| 2 | **모델 최적화 (학습)** | Early Stopping·학습 로그·결과 대시보드로 학습 효율화 | "AI를 똑똑하게 훈련시키고 자원 낭비 차단" |
| 3 | **에이전트 배포 절차** | 에이전트를 API Endpoint로 배포(에이전트 선택→옵션→배포) | "만든 AI를 실제 서비스로 출시" |
| 4 | **모델 배포 절차** | sLLM+LLM(GPT/Claude/Gemini/Llama 등) 선택 → 리소스(CPU/GPU/Memory)·Replica·양자화 설정 → Self-hosting/Serverless 배포 | "엔진을 골라 서버에 올리기" |
| 5 | **모델 버전 관리·검색 카탈로그** | 모델 버전/이력 기반 검색 | "어느 버전을 쓰는지 추적" |
| 6 | **GPUaaS — GPU Pool 관리** | GPU 자원을 풀(Pool)로 묶어 프로젝트별 배분·모니터링 | "비싼 GPU를 나눠 쓰는 관제" |
| 7 | **GPU 모니터링 주요화면** | Cluster Usage·Project별 사용량·Heatmap·Pending 요청 | "GPU가 어디서 얼마나 쓰이는지 실시간 현황판" |

**핵심 메시지:** "AI의 엔진(모델)과 연료(GPU)를 등록·훈련·배포·감시하는 운영실. 정보보안·인프라팀의 영역."

## 4. 우리 고정자산관리 프로토타입에 어떻게 적용할까 (설계)

### 4-1. 먼저 솔직한 선긋기 (전문 팀장의 조언)

우리 프로젝트는 **"운영 시스템"이 아니라 "시연용 데모"** 입니다 (HTML/CSS/JS + 샘플데이터, 백엔드는 Supabase 정도). 그러니:

- ✅ **적용할 가치 큼 (사용자가 보는 UX 개념)** → 이미지 ①·② 의 *화면·흐름 컨셉*
- ⚠️ **컨셉만 차용 (실제 구현 X)** → Datumo Eval 평가 대시보드, Agent 모니터링
- ❌ **적용 불필요 (인프라 영역)** → 이미지 ③ LLM Ops·GPU 관리 (시연에 안 보임, 만들 이유 없음)

> 핵심: **"진짜 AI 인프라를 만드는 게 아니라, 그렇게 동작하는 것처럼 보이는 시연 화면을 만든다."** 우리 AI Agent 화면이 이미 시나리오 기반으로 그렇게 동작하고 있음 → 이걸 제안서 컨셉으로 고급스럽게 확장.

### 4-2. 우리 자산관리에 맞춘 "멀티 에이전트" 재해석

제안서의 추상적 AI들을 **우리 업무로 번역**하면:

| 제안서 개념 | 우리 자산관리 버전 (시연용) |
|------------|---------------------------|
| 여러 전문 Agent | **노후자산 진단 Agent / 보험·계약 만료 점검 Agent / 예산영향 분석 Agent / 보고서 작성 Agent** |
| Agent 도구(MCP/Tool) | 자산 DB 조회, 보험계약 조회, 감가상각 계산, 지역별 현황 조회 |
| Agent 노드 전달 | "보험 만료 점검 → 예산영향 분석 → 보고서 작성"으로 결과 바통 터치 (우리 시나리오2와 동일 흐름) |
| Datumo Eval 평가 | AI 답변 "신뢰도 점수" 배지 (근거 자산 N건, 정책 기준 일치 등) |
| FLOW UI 빌더 | 자산관리 자동화 흐름을 블록으로 보여주는 "동작 과정 시각화" |

### 4-3. 단계별 적용 계획 (작게 쪼갬)

**Phase 1 — AI Agent 화면 "멀티 에이전트"화 (가장 효과 큼, 추천 1순위)**
- 현재: 질문→분석 6단계→답변 (단일 에이전트 느낌)
- 개선: 분석 패널에 **"어느 전문 Agent가 일하는 중"** 표시 (예: 🔍 보험점검 Agent → 💰 예산분석 Agent → 📄 보고서 Agent 순차 점등)
- 효과: 제안서의 "Multi Agent 협업"을 우리 화면에서 눈으로 보여줌. 코드 변경 작음(기존 6단계 라벨/아이콘 교체 수준).

**Phase 2 — AI 답변 신뢰도 배지 (Datumo Eval 컨셉)**
- 답변 말풍선에 "신뢰도 GOOD · 근거 자산 6건 · 정책 기준 일치" 같은 작은 평가 배지 추가.
- 효과: "AI 답을 검증한다"는 제안서 핵심 메시지를 1줄로 시연.

**Phase 3 — "AI 에이전트 빌더" 데모 화면 (FLOW UI 컨셉)**
- 사이드바에 신규 메뉴. 자산관리 자동화 흐름을 **블록 순서도**(드래그앤드롭 흉내)로 보여주는 정적/반응형 화면.
- 예: [자산조회] → [노후 판정] → [보험 확인] → [보고서] 블록 연결.
- 효과: 제안서의 No-Code/FLOW UI를 우리 도메인으로 시연. (구현량 중간)

**Phase 4 — Agent 관리/모니터링 데모 대시보드 (선택)**
- 어떤 Agent가 몇 번 실행됐는지, 신뢰도 평균 등 가짜 통계 대시보드.
- 효과: 제안서 "모니터링" 층까지 커버. (구현량 중간, 우선순위 낮음)

**제외:** LLM Ops/GPU/모델배포 화면 — 시연 가치 없음, 만들지 않음.

### 4-4. 추천 진행 순서

> **Phase 1 → 2 를 먼저** (적은 노력으로 제안서 핵심 두 가지 "멀티에이전트 협업"+"답변 검증"을 우리 화면에 즉시 반영). 반응이 좋으면 Phase 3(빌더 화면)로 확장.
